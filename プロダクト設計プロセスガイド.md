# プロダクト設計プロセスガイド

「ペルソナ → カスタマージャーニーマップ → ユースケース定義」を一貫して進めるためのガイド。 個人でAIと共同作業しながら初稿を作成し、相互レビューで品質を高める運用を前提とする。

---

## なぜこのプロセスが必要か

プロダクト開発では「何を作るか」を決める前に、「誰のために、どんな体験を提供するか」を明確にする必要がある。これが曖昧なまま開発に入ると、手戻りやスコープの膨張が発生し、チーム全体の生産性が落ちる。

このガイドでは、3つの成果物を段階的に作成する。それぞれの役割は以下の通り。

### ペルソナ — 「誰のために作るか」を定める

ペルソナは、プロダクトのターゲットユーザーを具体的な人物像として描いたものである。

**なぜ必要か**: チームメンバー全員が同じユーザー像を共有できていないと、機能の優先度判断や仕様の意思決定がブレる。「このペルソナならどう感じるか？」という共通の判断軸を持つことで、議論の収束が速くなる。

### カスタマージャーニーマップ — 「どんな体験を届けるか」を描く

カスタマージャーニーマップ（CJM）は、ペルソナがプロダクトに出会う前から定着するまでの体験全体を時系列で可視化したものである。

**なぜ必要か**: 個々の機能だけを見ていると、体験の「つなぎ目」が見落とされがちになる。CJMを通じて、ユーザーの感情の起伏やペインポイントを俯瞰することで、どこに注力すべきかの優先順位が明確になる。

### ユースケース定義 — 「何を作るか」を具体化する

ユースケース定義は、CJMで描いた体験を実現するために「プロダクトが何をできる必要があるか」を、実装可能な粒度で記述したものである。

**なぜ必要か**: ユースケース定義が十分な品質で作成されると、以下のことが可能になる。

- **UIメンバー** — 各ユースケースの基本フロー・代替フローを元に、画面設計やプロトタイプの作成に着手できる  
- **開発メンバー** — アクター、事前条件、事後条件、フローの情報を元に、技術的な見積もり（工数・難易度・リスク）に進められる  
- **QAメンバー** — 基本フローと代替フローがそのままテストケースの骨格になる  
- **PO / ビジネス側** — 優先度つきのユースケース一覧で、スコープとリリース計画の合意形成ができる

つまり、**ユースケース定義は「企画」と「実行（UI設計・開発・QA）」の橋渡しとなる成果物**である。ここの品質が低いと、UIメンバーや開発メンバーが「何を作ればいいか分からない」状態で手戻りが発生する。逆に、ここが丁寧に作られていれば、チームの各メンバーが自律的に動き出せる。

### 3つの成果物のつながり

```
ペルソナ（誰のため？）
  │  ペルソナの「フラストレーション」「現在の行動」が
  │  ジャーニーマップの課題・行動の根拠になる
  ▼
カスタマージャーニーマップ（どんな体験を？）
  │  CJMの各ステージの「改善の機会」「タッチポイント」が
  │  ユースケースの洗い出し根拠になる
  ▼
ユースケース定義（何を作る？）
  │  UIメンバー → 画面設計に着手
  │  開発メンバー → 見積もりに着手
  │  QAメンバー → テストケース作成に着手
  ▼
実行フェーズへ
```

**重要**: 3つの成果物は独立したドキュメントではなく、一貫した「ストーリー」として繋がっている。ペルソナで定義した人物が、ジャーニーマップの主人公であり、ユースケースの利用者である。どこかで矛盾があれば、上流に遡って修正する。

## 全体の流れ

```
Step 1            Step 2                    Step 3
ペルソナ設計  →  カスタマージャーニーマップ  →  ユースケース定義
 (誰のため?)      (どんな体験を?)              (何を作る?)
```

### 進め方: 個人作業 \+ 相互レビュー

```
作成者（個人）                    レビュアー
    │                              │
    ├─ 1. AIと共同でMarkdown初稿作成  │
    │     各Stepのセルフチェック実施   │
    │                              │
    ├─ 2. Markdown → 清書版に変換    │
    │                              │
    ├─ 3. 清書版をレビュー依頼  ───→ │
    │                              ├─ 4. 清書版にコメントで指摘
    │  ←─────────────────────────── │
    │                              │
    ├─ 5. 指摘をMarkdownに反映       │
    │     （AIに修正依頼してもOK）    │
    │     清書版を再変換             │
    │                              │
    └─ 6. 必要ならStep 3〜5を繰り返し │
```

**ポイント**:

- **各Stepは1人で完結できる**。AIとの対話で初稿を作り、各Stepのチェックポイントで自己チェックする  
- **清書版に変換してからレビュー依頼**する。Markdownのまま渡さない（レビュアーがコメントをつけづらい）  
- **レビューコメントのフォーマット**を統一しておくと、やり取りがスムーズになる:

```
[MUST]   修正しないと次に進めない問題
[SHOULD] 品質向上のために修正すべき
[NIT]    細かい改善提案。対応は任意
```

## Step 1: ペルソナ設計

### 目的

「誰のために作るのか」を明確にし、レビュアーや関係者と共有できる、具体的で共感可能な人物像を作る。

### インプットに必要なもの

| インプット | 理想 | 最低限（仮説ベース） |
| :---- | :---- | :---- |
| ユーザーインタビュー | 各セグメント3〜5名のインタビュー記録 | 自分が持っている顧客理解・ドメイン知識 |
| 定量データ | アンケート結果、アクセス解析 | 市場調査レポート、競合アプリのレビュー分析 |
| セグメント仮説 | アフィニティマッピングから抽出 | 自分の仮説をAIとの壁打ちで整理 |

### 手順

#### 1-1. セグメントの仮説を立てる

- ターゲットユーザーの中で、**異なるニーズ・行動パターン・動機**を持つグループを2〜3つ特定する  
- 各セグメントに「〜のために〜したい〜な人」という一文で名前をつける

**どうやってセグメントを分けるか — 判断基準**:

以下の3つの軸のうち、少なくとも2つで明確な差がある場合、別セグメントとして分ける。

| 軸 | 問いかけ | 例（KoeMemo） |
| :---- | :---- | :---- |
| **動機（なぜ使うか）** | このプロダクトに何を期待しているか？ | 効率化 vs 感情整理 vs 新鮮な体験 |
| **行動パターン** | 現状どうやって課題を解決しようとしているか？ | Notionで書く vs LINEに吐き出す vs 色々なアプリを試す |
| **制約条件** | 利用を妨げている最大の障壁は何か？ | 時間がない vs 言語化が苦手 vs 飽きやすい |

**判断に迷ったときのテスト**: 「この2人に同じ機能を見せたとき、反応が違うか？」を想像する。反応が同じなら1つのセグメントにまとめてよい。

**Good/Bad例（KoeMemoの場合）**:

Bad — 属性だけで分けている:

```
- 「20代の独身男性」
- 「30代の既婚女性」
- 「40代の管理職」
→ 年齢・性別で分けても、日記アプリの使い方は予測できない
```

Good — 動機と行動パターンで分けている:

```
- 「時間がないけど自己成長したい効率重視の若手ビジネスパーソン」
   → 動機: 効率化、行動: Notionで挫折、制約: 時間
- 「気持ちの整理手段を探しているストレスフルな中堅社員」
   → 動機: 感情整理、行動: LINEに独り言、制約: 言語化力
- 「新しい記録体験を求める習慣化チャレンジャー」
   → 動機: 新鮮さ、行動: アプリを転々、制約: 飽きやすさ
→ 3者にKoeMemoを見せたら、注目するポイントが明確に異なる
```

#### 1-2. ペルソナの骨格を作る

**各項目で何を書くべきか — 判断基準**:

| 項目 | 書くべきこと | よくある間違い |
| :---- | :---- | :---- |
| **基本情報** | プロダクトの利用文脈に影響する属性だけ | 関係ない属性まで盛り込む（血液型、趣味の詳細等） |
| **背景ストーリー** | **プロダクトが解決する課題に至るまでの文脈**。1日の流れの中で「いつ・どこで・なぜ」困っているか | 人生のあらすじになってしまう（幼少期から書き始める等） |
| **ゴール** | プロダクトを使って**具体的に**達成したいこと | 「幸せになりたい」のような抽象的な願望 |
| **フラストレーション** | **今この瞬間**感じている不満・苦痛。感情を伴う表現で | 機能要望の列挙（「検索機能がほしい」等） |
| **現在の行動** | 課題に対して**今やっている代替手段** | 理想の行動を書いてしまう |

**Good/Bad例 — フラストレーション**:

Bad — 機能要望になっている:

```
- 音声入力機能がほしい
- AIで文章を整形してほしい
- リマインド通知がほしい
```

→ これはペルソナではなく要件定義。ペルソナはプロダクトの存在を知らない前提で書く

Good — 今感じている不満を感情込みで:

```
- 白紙のエディタを前にすると、何を書けばいいかわからずフリーズする
- 文章を打つ時間と気力がない（寝る前はスマホを見るのが限界）
- 過去に3回日記アプリをインストールしたが、いずれも1週間で離脱した
```

→ 読んだ人が「あー、わかる」と共感できる。プロダクト非依存の課題記述

各セグメントについて、以下を埋める:

```
## ペルソナN: [アーキタイプ名]

### 基本情報
| 項目 | 内容 |
|------|------|
| 氏名 | [日本語のフルネーム] |
| 年齢 | [具体的な年齢] |
| 職業 | [具体的な職種] |
| 居住地 | [具体的な地域] |
| 家族構成 | [具体的に] |
| ITリテラシー | [高/中/低 + 具体的な使用ツール] |

### 人物写真のイメージ
[どんな外見・雰囲気の人物かを文章で描写]

### キャッチーな一言
> 「[その人の価値観やライフスタイルを象徴する一言]」

### 自己紹介 / 背景ストーリー
[その人の1日の様子や、サービスの領域に関連する悩みが伝わる短い物語]

### ゴール（Goals）
- [サービス利用を通じて達成したいこと 3つ]

### フラストレーション（Frustrations）
- [現状の課題、悩み、不満 3つ]

### 現在の行動・利用ツール
- [課題解決のために現在どう行動しているか 3つ]
```

#### 1-3. 優先度をつける

| 優先度 | 説明 | 目安 |
| :---- | :---- | :---- |
| **Primary** | プロダクトのコア価値と最も合致するペルソナ | 1名 |
| **Secondary** | 差別化要素を活かせるペルソナ | 1名 |
| **Tertiary** | 検証・拡張に有用なペルソナ | 0〜1名 |

**Primaryをどう決めるか — 判断基準**:

以下の3問に最も「はい」が多いペルソナがPrimary:

1. このペルソナの課題を解決できなかったら、プロダクトの存在意義がなくなるか？  
2. このペルソナの満足度が高ければ、他のセグメントにも波及するか？  
3. このペルソナが最も多い（市場が大きい）か？

**KoeMemoでの判断例**:

佐藤翔太（効率重視）をPrimaryにした理由:

1. 「空白恐怖の解消 \+ 音声入力による時短」はKoeMemoのコア価値そのもの → はい  
2. 効率系ユーザーが「便利だ」と評価すれば、感情整理系にも「簡単に使える」と訴求できる → はい  
3. 「日記を試したが続かなかった社会人」は最も母数が大きい → はい

### チェックポイント

- [ ] 各ペルソナは実在しそうな具体性があるか？（名前、年齢、職業が具体的）  
- [ ] ゴールとフラストレーションが、プロダクトのコア価値と結びついているか？  
- [ ] 「現在の行動」に、プロダクトが代替する行動が含まれているか？  
- [ ] ペルソナ同士で、ニーズ・行動パターンが明確に異なるか？  
- [ ] 第三者が読んでも「この人知ってる」と感じるリアリティがあるか？

### よくある失敗パターン

| 失敗 | 症状 | 対策 |
| :---- | :---- | :---- |
| **デモグラフィックだけのペルソナ** | 年齢・性別・職業は書いてあるが、行動や感情が薄い | 背景ストーリーとフラストレーションを最も時間をかけて書く |
| **理想ユーザーペルソナ** | 「プロダクトを完璧に使いこなす人」になっている | フラストレーションと「離脱しそうなポイント」を意識的に書く |
| **全員同じペルソナ** | セグメントが違うのに、ゴールやフラストレーションが似通っている | 「この人はAの機能を使うが、Bの機能には興味がない」という差分を明確にする |
| **データの裏付けがない** | 自分の想像だけで作ってしまう | 仮説ベースでも「検証予定」と明記し、後でインタビューで更新する |

## Step 2: カスタマージャーニーマップ

### 目的

ペルソナが「プロダクトに出会う前〜定着するまで」の体験全体を可視化し、課題と改善機会を特定する。

### インプットに必要なもの

| インプット | 入手元 |
| :---- | :---- |
| 完成したペルソナ | Step 1の成果物 |
| プロダクトの機能一覧 | PRD / 機能仕様書 |
| 競合プロダクトの体験 | 競合分析（自分で使ってみた体験がベスト） |

### 手順

#### 2-1. As-Is（現状）ジャーニーを作る

**なぜ先にAs-Isを作るのか**: プロダクトが解決する「現状の課題」を深く理解しないと、To-Beジャーニーで「何がどう良くなるのか」を説得力を持って描けない。

```
## As-Is ジャーニーマップ

**対象ペルソナ**: [ペルソナ名]
**シナリオ**: [プロダクトが解決する課題に直面する状況]
**ゴール**: [この領域でユーザーが本来達成したいこと]

### 感情曲線（テキスト表現）
[感情の起伏をASCIIアートで表現]

| ステージ | ユーザーの行動 | タッチポイント | 思考・感情 | 課題・ペインポイント | 改善の機会 |
|---------|-------------|--------------|----------|-------------------|----------|
| [ステージ名] | [具体的な行動] | [接点] | [思考と感情 (+/-)] | [困っていること] | [解決のアイデア] |
| ... | ... | ... | ... | ... | ... |
```

**ステージの決め方 — 判断基準**:

ゴールに向かう一連の行動を時系列で並べ、「心理状態が切り替わるポイント」で区切る。

具体的には、以下のどれかが変わったら「ステージの境目」:

| 変化の種類 | 問いかけ | 例（KoeMemo As-Is） |
| :---- | :---- | :---- |
| **感情の転換** | ポジティブ↔ネガティブが切り替わるか？ | やる気満々(動機付け) → 選択疲れ(ツール選び) |
| **行動の質的変化** | やっていることの種類が変わるか？ | 情報収集(検索) → 実際に書く(初回挑戦) |
| **意思決定** | 何かを決めたか/諦めたか？ | 「Notionでいいか」と決定 → 「向いてない」と離脱 |

**ステージは5〜8個が適切**。3個以下だと粗すぎて課題が見えず、10個以上だと細かすぎて全体像を見失う。

**Good/Bad例 — ステージの切り方**:

Bad — 粗すぎる:

```
認知 → 利用 → 定着
```

→ 「利用」の中に初回体験・2日目・1週間が全部入っている。初回と1週間後では感情が全く違うのに、同じステージに押し込んでいる

Bad — 細かすぎる:

```
App Store検索 → 検索結果閲覧 → スクショ確認 → レビュー確認 → DLボタンタップ → DL完了待ち → 初回起動 → ...
```

→ UI操作の手順書になっている。心理状態は「興味 → 評価中」で一貫しているので1ステージでよい

Good — 心理の変化で区切る:

```
動機付け(やる気) → ツール選び(迷い) → 初回挑戦(期待→挫折) → 2〜3日目(惰性) → 1週間(罪悪感) → 離脱(諦め)
```

→ 各ステージで感情が異なり、それぞれ固有のペインポイントがある

典型的なステージ例:

```
動機付け → 手段探し → 初回挑戦 → 継続試行 → 壁にぶつかる → 離脱/定着
```

#### 2-2. To-Be（プロダクト利用時）ジャーニーを作る

As-Isの各ペインポイントに対して、「プロダクトがどう解決するか」を具体的に描く。

```
## To-Be ジャーニーマップ

**対象ペルソナ**: [ペルソナ名]
**シナリオ**: [プロダクトを発見し、利用し始める状況]
**ゴール**: [プロダクトを通じて達成すること]

| ステージ | ユーザーの行動 | タッチポイント | 思考・感情 | 課題・ペインポイント | 改善の機会 |
|---------|-------------|--------------|----------|-------------------|----------|
| [ステージ名] | [具体的な行動] | [プロダクトの画面や機能] | [思考と感情 (+/-)] | [残る課題] | [さらなる改善余地] |
| ... | ... | ... | ... | ... | ... |
```

**To-Beでも課題を書くのが重要**: プロダクトを使っても完璧にはならない。残る課題を正直に書くことで、次の改善サイクルのインプットになる。

**To-Beの各列で何を書くか — 判断基準**:

| 列 | As-Isとの違い | やりがちなミス |
| :---- | :---- | :---- |
| **タッチポイント** | 具体的な画面名・機能名で書く | 「アプリ」とだけ書く |
| **思考・感情** | ポジティブだけでなく「不安」「戸惑い」も正直に | 全ステージがポジティブ |
| **課題・ペインポイント** | プロダクトを使っても**残る**課題を書く | 空欄にしてしまう |
| **改善の機会** | 次のバージョンで取り組む具体策 | 「改善する」のような曖昧な記述 |

**Good/Bad例 — To-Beのペインポイント列**:

Bad — 空欄 or 「特になし」:

```
| 初回体験 | ... | ... | 「楽しい！」(++) | 特になし | ... |
```

→ 初回で全員が感動する想定は楽観的すぎる

Good — プロダクトを使っても残る現実的な課題:

```
| 初回体験 | ... | ... | 「おっ、楽しい」(+) | 最初の一言を発するまでの緊張。音声認識の精度への不安 | オンボーディングで「気軽に話してみてください」とガイド |
```

→ 「楽しい」の前にある「一瞬の不安」まで捉えている

#### 2-3. As-Is / To-Be 比較サマリーを作る

```
## 比較サマリー

| 観点 | As-Is | To-Be | プロダクトが提供する価値 |
|------|-------|-------|----------------------|
| [比較軸] | [現状] | [改善後] | [具体的な価値] |
```

### チェックポイント

- [ ] ペルソナの「フラストレーション」がAs-Isジャーニーのペインポイントに反映されているか？  
- [ ] To-Beジャーニーで、As-Isの主要ペインポイントが解消されているか？  
- [ ] 感情曲線に明確な起伏があるか？（ずっとポジティブは不自然）  
- [ ] To-Beジャーニーにも課題・ペインポイントが書かれているか？  
- [ ] 各ステージのタッチポイントが、実際のプロダクトの画面/機能と対応しているか？  
- [ ] 「改善の機会」が具体的なアクションに落とし込めるレベルか？

### よくある失敗パターン

| 失敗 | 症状 | 対策 |
| :---- | :---- | :---- |
| **ハッピーパスだけのTo-Be** | プロダクト利用時にネガティブな感情が一切ない | 「1週間の壁」「飽き」「エラー時」等の現実的な課題を必ず入れる |
| **ステージが粗すぎる** | 「認知→利用→定着」の3段階だけ | 心理状態の変化ポイントでさらに細分化する（初回と2回目は全く異なる体験） |
| **タッチポイントが曖昧** | 「アプリ」としか書いていない | 具体的な画面名や機能名で記述する（例：「録音画面の話題カード」） |

## Step 3: ユースケース定義

### 目的

ジャーニーマップの体験を実現するために「プロダクトが何をできる必要があるか」を、実装可能な粒度で定義する。

### インプットに必要なもの

| インプット | 入手元 |
| :---- | :---- |
| 完成したペルソナ | Step 1 |
| To-Beジャーニーマップ | Step 2 |
| 技術制約 | エンジニアとの相談 |

### 手順

#### 3-1. アクターを定義する

```
| アクター | 説明 | 該当ペルソナ |
|---------|------|------------|
| [アクター名] | [役割の説明] | [対応するペルソナ名] |
```

**内部アクター（システム）も忘れずに**: AIや外部APIなど、ユーザーの操作なしに動くアクターも定義する。

#### 3-2. ユースケース一覧を洗い出す

To-Beジャーニーの各ステージから、必要な機能をユースケースとして抽出する。

```
| カテゴリ | ID | ユースケース名 | アクター | 優先度 |
|---------|-----|--------------|---------|-------|
| [カテゴリ] | UC-XX | [動詞 + 目的語] | [アクター名] | Must/Should/Could |
```

**命名規則**: 「〜する」の形式で、アクターの視点から記述する。

**Good/Bad例 — ユースケースの命名**:

Bad — 機能名・技術用語になっている:

```
| 日記作成 | UC-10 | 音声認識機能 | ... |
| 日記作成 | UC-11 | カード表示機能 | ... |
| 日記作成 | UC-13 | 日記生成API | ... |
```

→ エンジニアの設計書になっている。「誰が何のために使うか」が見えない

Good — ユーザーの行動として書かれている:

```
| 日記作成 | UC-10 | 音声で日記を録音する | 一般ユーザー | Must |
| 日記作成 | UC-11 | 話題カードに沿って話す | 一般ユーザー | Must |
| 日記作成 | UC-13 | AIに日記を生成してもらう | 一般ユーザー | Must |
```

→ 主語がユーザー。ペルソナの行動としてイメージできる

**ユースケースの粒度 — 判断基準**:

適切な粒度の目安: **「1回の操作セッションで完結する、1つの目的」** \= 1ユースケース。

| 粒度テスト | 問いかけ | 判定 |
| :---- | :---- | :---- |
| **大きすぎないか？** | この中に「一旦止めて、後で再開する」ポイントがあるか？ | あるなら分割 |
| **小さすぎないか？** | これ単体で、ユーザーの目的が達成されるか？ | されないなら統合 |
| **独立しているか？** | このユースケースだけ削除しても、前後のフローに影響しないか？ | するなら別UCの一部かも |

**Good/Bad例 — ユースケースの粒度**:

Bad — 大きすぎる:

```
UC-10: 日記を作成する
  1. 録音画面を開く → 2. 話題カードが表示される → 3. 話す → 4. AI分析
  → 5. 完成度確認 → 6. 日記生成 → 7. プレビュー → 8. 編集 → 9. 保存
```

→ 9ステップ。録音と生成と保存は独立した目的。途中で中断するポイントもある

Good — 目的ごとに分割:

```
UC-10: 音声で日記を録音する（目的: 音声素材の収集）
UC-13: AIに日記を生成してもらう（目的: 構造化された日記の取得）
UC-15: 日記を保存する（目的: データの永続化）
```

→ それぞれ独立した目的がある。UC-13だけ改善しても他に影響しない

Bad — 小さすぎる:

```
UC-10a: 録音ボタンをタップする
UC-10b: マイク権限を許可する
UC-10c: 話す
UC-10d: 録音を停止する
```

→ UI操作手順になっている。これらは全てUC-10の基本フロー内のステップ

**優先度の基準**:

| 優先度 | 基準 |
| :---- | :---- |
| **Must** | これがないとPrimaryペルソナのコアシナリオが成立しない |
| **Should** | UXを大きく向上させるが、なくてもコアシナリオは成立する |
| **Could** | あると良いが、初期リリースでは不要 |

**優先度の判断 — 迷ったときの問いかけ**:

Must か Should かで迷ったら:

「この機能なしで、Primaryペルソナ（佐藤翔太）が**初回から1週間**使い続けられるか？」 → 使えない → Must → 不便だが使える → Should

Should か Could かで迷ったら:

「初回リリースのユーザーレビューで、この機能がないことに不満が出るか？」 → 出る → Should → 出ない → Could

#### 3-3. 主要ユースケースの詳細を書く

Must優先度のユースケースについて、以下のフォーマットで詳細化する:

```
### UC-XX: [ユースケース名]

**概要**: [1〜2文で何をするユースケースかを説明]

| 項目 | 内容 |
|------|------|
| **アクター** | [主アクター名] |
| **トリガー** | [何がきっかけで開始されるか] |
| **事前条件** | [開始時に満たしている必要がある条件] |
| **事後条件** | [正常完了時に達成される状態] |

**基本フロー（メインシナリオ）**:
1. [ステップ1]
2. [ステップ2]
...

**代替フロー**:
- **Xa. [条件]の場合**:
  1. [ステップ]
```

#### 3-4. ペルソナ × ユースケースのマッピング

各ペルソナがどのユースケースを重視するかを整理する。

```
| ユースケース | ペルソナ1 | ペルソナ2 | ペルソナ3 |
|------------|:---------:|:---------:|:---------:|
| UC-XX | [重要度 + 理由] | [重要度 + 理由] | [重要度 + 理由] |
```

### チェックポイント

- [ ] To-Beジャーニーの全ステージがユースケースでカバーされているか？  
- [ ] ユースケース名がアクターの視点で書かれているか？（「〜する」形式）  
- [ ] Must優先度のユースケースだけでPrimaryペルソナのコアシナリオが成立するか？  
- [ ] 代替フロー（エラー、例外）が考慮されているか？  
- [ ] 各ユースケースの事前条件・事後条件が明確か？  
- [ ] ペルソナ × ユースケースのマッピングで、ペルソナ間の差異が見えるか？

### よくある失敗パターン

| 失敗 | 症状 | 対策 |
| :---- | :---- | :---- |
| **機能一覧になっている** | 「音声認識機能」「カード表示機能」のように書いてしまう | 必ず「ユーザーが〜する」の視点で書き直す |
| **粒度がバラバラ** | 「ログインする」と「アプリを使う」が同列 | 1つのユースケース \= 1つの具体的な目標達成 |
| **代替フローがない** | 基本フローだけで、エラーや例外が未考慮 | 「もし〜だったら？」を各ステップで問いかける |
| **全部Must** | 優先度をつけられていない | 「これがなかったらPrimaryペルソナは使えるか？」で判断 |

## フォーマット戦略

### 基本方針: Markdown → 清書版の2層構造

```
[作業レイヤー]                    [提出・合意レイヤー]
Markdown (.md)  ──── 変換 ────→  清書版（レビュー・顧客提示用）
  │                                │
  │ AIで高速に生成・修正            │ ビジュアル・レイアウトが整った状態
  │ Gitで版管理                    │ コメントで相互レビューしやすい
  │ 差分比較が容易                  │ 顧客への説明に使える
  │                                │
  └── 常に最新（マスターデータ）    └── レビュー依頼・顧客提示のタイミングで作成
```

**なぜ2層にするのか**:

- AIでの生成・修正はMarkdownが圧倒的に速い（清書ツール上で直接編集すると時間がかかる）  
- 一方、Markdownのままでは顧客提示に耐えず、レビューコメントもつけづらい  
- Markdownをマスターにしておけば、レビュー指摘の反映もAIに依頼できる

### 成果物別の清書フォーマット

| 成果物 | 作業用 | 清書版 | 清書版を使う理由 |
| :---- | :---- | :---- | :---- |
| **ペルソナ** | Markdown | **Googleスライド** | 1ペルソナ1ページのカード型が視覚的。写真イメージも配置しやすい。スライド単位でレビューコメントをつけやすい |
| **CJM** | Markdown | **Miro** | ステージの横並び表示と感情曲線の可視化が得意。付箋でレビューコメントをつけやすい |
| **ユースケース定義** | Markdown | **Googleスプレッドシート** | フィルタ・ソートで優先度別に閲覧可能。セル単位でレビューコメントをつけやすい |
| **プロセスガイド** | Markdown | **Markdown（このファイル自体）** | 社内ナレッジとして長期運用。エンジニアもデザイナーも参照するため、Gitで版管理 |

#### 変換手順

2. Markdownの各セクションをスライドの対応エリアにコピー  
3. 「背景ストーリー」はスライド用に2〜3行に要約する（詳細はMarkdownを参照）  
4. 人物写真のイメージに合うフリー素材を探して配置  
5. ペルソナの優先度（Primary / Secondary / Tertiary）をラベルで表示

**注意**: スライドは「一目で人物像をつかむ」ためのもの。情報を詰め込みすぎない。詳細はMarkdownを正とする。

#### 変換手順

1. Miroでテンプレートボードを開く（またはテンプレートをコピー）  
2. ステージ数に合わせて列を追加/削除  
3. Markdownのテーブルから各セルの内容をコピー  
4. 感情曲線を折れ線で描画（Miroの線ツール使用）  
5. ペインポイントは赤系、改善機会は緑系の付箋で色分け  
6. As-Is と To-Be は別フレームにし、横に並べて比較できるようにする

### ユースケース定義 → Googleスプレッドシート

#### シート構成（3シート）

**シート1: ユースケース一覧**

| A列 | B列 | C列 | D列 | E列 | F列 | G列 |
| :---- | :---- | :---- | :---- | :---- | :---- | :---- |
| カテゴリ | ID | ユースケース名 | アクター | 優先度 | CJMステージ対応 | ステータス |

- 「優先度」列にデータバリデーション（Must / Should / Could）を設定  
- 「ステータス」列に（未着手 / レビュー中 / 承認済み）を設定  
- フィルタビューで「Must のみ」「レビュー中のみ」を素早く切り替え可能に

**シート2: ユースケース詳細**

| A列 | B列 | C列 | D列 | E列 | F列 | G列 | H列 |
| :---- | :---- | :---- | :---- | :---- | :---- | :---- | :---- |
| ID | ユースケース名 | 概要 | アクター | トリガー | 事前条件 | 事後条件 | 基本フロー |

- 「基本フロー」列はセル内改行で番号付きステップを記述  
- 代替フローは行を追加して ID を「UC-10a」「UC-10b」のように枝番で管理

**シート3: ペルソナ × ユースケース マッピング**

| A列 | B列 | C列 | D列 | E列 |
| :---- | :---- | :---- | :---- | :---- |
| ID | ユースケース名 | ペルソナ1 重要度 | ペルソナ2 重要度 | ペルソナ3 重要度 |

- 重要度を「最重要 / 重要 / 低」の3段階で記入  
- 条件付き書式で色分け（最重要=赤、重要=黄、低=グレー）

#### 変換手順

1. Googleスプレッドシートでテンプレートファイルを開く（またはコピー）  
2. Markdownのユースケース一覧テーブルをシート1にコピー  
3. 詳細ユースケースの内容をシート2の各列に転記  
4. ペルソナ × ユースケースマッピングをシート3に転記  
5. データバリデーション、フィルタビュー、条件付き書式を設定

**注意**: スプレッドシートはセル単位でコメントがつけられるため、「このユースケースの事前条件が足りない」等のピンポイントなレビューが容易。

## AIとの共同作業のコツ

### プロンプトに含めるべき情報

| 情報 | 目的 |
| :---- | :---- |
| **プロダクトの概要** | AIにドメイン知識を与える |
| **ターゲットユーザーの特徴** | ペルソナの方向性を定める |
| **解決したい課題** | ペインポイントの核をぶらさない |
| **競合との差別化ポイント** | 独自の価値提案を反映させる |
| **既存のリサーチデータ** | データに裏付けられた具体性を持たせる |

### AIへの効果的な指示の出し方

```
具体的に指示する:
  ✕ 「ペルソナを作って」
  ○ 「30代の一人暮らし会社員で、日記を始めたいが続かない人のペルソナを、
      背景ストーリーとフラストレーションを特に詳しく作ってください」

段階的に深める:
  1回目: 骨格を生成
  2回目: 「背景ストーリーをもっと具体的に」
  3回目: 「フラストレーションにXXのインタビュー結果を反映して」
  4回目: 「全体を通して整合性を確認して最終版にして」
```

### AIに任せて良いこと / 人間がやるべきこと

| AIに任せて良い | 人間がやるべき |
| :---- | :---- |
| フォーマットに沿った構造化 | セグメント仮説の設定 |
| 背景ストーリーの肉付け | リサーチデータとの照合 |
| 代替フローの網羅的な洗い出し | 優先度の最終判断 |
| 感情曲線の推定 | レビュー指摘への対応判断 |
| ユースケース間の整合性チェック | ビジネス上の意思決定 |
